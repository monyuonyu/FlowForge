# はじめてのチュートリアル（FlexSim風 離散事象シミュレータ）

このドキュメントは、本ツールを **約10分** で一通り使えるようになるための入門です。
後半では **同梱サンプル** を読み込みながら、シミュレーションの実測値が **待ち行列理論の
理論値と一致する** ことを確かめます（＝「この数字は信用できる」と自分の目で確認できます）。

- 対象バージョン: Godot 4.7.1 / GDScript（イベント駆動DES）
- 前提知識: 特になし（待ち行列理論を知らなくても、理論値は本書と各サンプルに書いてあります）

---

## 0. このツールは何か（30秒）

モノ（アイテム）が **Source（発生）→ Queue（待ち行列）→ Processor（加工）→ Conveyor（搬送）
→ Sink（消滅）** と流れていく工場・物流ラインを、3Dで **可視化・編集・実験** できる
離散事象シミュレーション（DES）です。

3つの性質を覚えておくと理解が早いです。

1. **イベント駆動**: 時間を細かく刻むのではなく「次に何かが起きる時刻」へ一気にジャンプします。
   だから8時間分の操業を一瞬で回せます。
2. **決定論的（再現可能）**: 乱数はシード管理された独立ストリーム。**同じ seed なら毎回
   まったく同じ結果**（ビット単位で一致）。A/B比較や検証の土台になります。
3. **理論と整合**: 標準的な待ち行列モデル（M/M/1, M/M/c など）を組むと、実測KPIが
   閉じた式の理論値に収束します（本書で確認します）。

## 1. 起動する（1分）

1. [Godot 4.x](https://godotengine.org/) を起動し、「インポート」から本フォルダの
   `project.godot` を選ぶ。
2. `F5`（実行）。既定モデル（Source→Queue→Machine→Conveyor→…→Sink、作業者2名）が開きます。
3. 左上 **「▶ 開始」** を押すとアイテムが流れ始めます。もう一度押すと一時停止。

> ヘッドレス（画面なし）でエンジンの自己検査だけ回すこともできます（後述の第7節）。

## 3. 実行モードと編集モード（2分）

ツールには2つのモードがあります。

- **実行モード（既定）**: シミュレーションを回してKPIを見るモード。右パネルに稼働率や
  ガント、下にコンソール・時系列・ヒストグラムが出ます。
- **編集モード**: 左上「🔧 編集モード」をONにすると、モデルを組み替えるモード。
  実行専用パネルは自動的に隠れ、右上に **インスペクタ** が出ます。

### 編集モードでできること
- **追加**: 編集ツールバーの `Source / Queue / Rack / Processor / Conveyor / Sink /
  Combiner / Separator` ボタン → 地面クリックで設置。
- **選択**: オブジェクトをクリック。インスペクタで名前・座標・回転・縮尺を編集。
- **配線**: 「🔗 配線」ON → 元→先の順にクリック（またはインスペクタの「接続」で選択）。
  設備間に矢印が表示されます。
- **パラメータ**: インスペクタの「パラメータ (JSON)」欄で分布や容量を編集し「適用」。
  例）Processor の加工時間を指数分布・平均2.0に: `{"process_time":{"type":"exp","a":2.0}}`
- **スクリプト**: 各設備に GDScript（`extends LogicBase`）を割り当ててロジックを拡張可能。
- **作業者/搬送者**: ツールバーの ＋/－ で増減。作業者は共有資源として奪い合われます。

編集が済んだら「🔧 編集モード」をOFFに戻し、「▶ 開始」で回します。

## 4. 実験を回す（2分）

単発の実行は見て楽しいですが、**乱数を含むモデルの正しい評価には複数回の反復と
立ち上がり除外が必要** です。実験バーの各欄がそれです。

- **種 (seed)**: 乱数の種。同じ種＝同じ結果。
- **warm (warmup)**: 立ち上がり時間。この時刻で統計をリセットし、空の系が埋まるまでの
  過渡期を除外します（定常状態の評価に必須）。
- **反復 (reps)**: 独立反復回数。
- **長さ (run_len)**: 1回あたりのシミュレーション時間。

**「🧪 実験」** を押すと、指定条件で瞬時に reps 回まわし、スループットと滞留時間の
**平均 ± 95%信頼区間** をコンソールへ出力、`user://experiment.csv` に保存します。

そのほか:
- **🅰🅱 シナリオ実験(A/B)**: 到着間隔を変えた2条件を共通乱数法(CRN)で比較。
- **🧭 シナリオ掃引**: 選択オブジェクトの1パラメータをカンマ区切りの値リストで掃引。
- **🎯 最適化**: 選択パラメータをグリッド探索し、スループット最大の値を探す。
- **📄 レポート出力**: HTML+CSV のレポートを生成（パスはコンソールに表示）。

## 5. KPIの読み方（1分）

右上のKPIタイルと設備行の意味:

| KPI | 意味 | 対応する理論量 |
|-----|------|----------------|
| 産出(個) | Sink に到達した総数 | 完了数 |
| スループット(個/時) | 単位時間あたり産出。`Sink.total / 経過 × 3600` | λ_eff（実効処理率） |
| 平均滞留(秒) | アイテムが系内に居た平均時間 | W（滞在時間）|
| 仕掛(現在) | いま系内にいる数 | WIP（瞬間値） |
| 仕掛(時間平均) | WIP の時間加重平均 | L（平均系内数） |
| 空き作業者 | 手待ちの作業者数 | 資源の空き |
| 設備行 `Lq / Wq` | Queueの平均行列長・平均待ち時間 | Lq, Wq |

**リトルの法則** `L = λ · W`（仕掛 = スループット × 滞留）が数値的に成り立つことも
確認できます。

## 6. サンプルで理論値を確かめる（📚 サンプル）

ここが本書の核心です。左上4段目の **「📚 サンプル」** ドロップダウンから任意の
サンプルを選ぶと、

1. そのモデルが読み込まれ（既存の読込パスと同じ。seed/warmup もモデル値に設定）、
2. コンソールに **タイトル・概要・期待値（理論値）** が出ます。

つまり「回す前に、期待すべき数字」がすぐ分かります。あとは **「▶ 開始」**（または
「🧪 実験」）で実測し、理論値に収束することを確かめてください。長期実行＋warmup を
推奨します（定常状態に達するまで）。

以下、代表4本のウォークスルー。**理論値**と、同梱の自己検査（第7節）で実測された値を
併記します。数字が合っていることが「信用できる」証拠です。

### 6.1 M/M/1 単一窓口待ち行列（`mm1_queue`）
- 構成: Source → Queue → Processor(1台) → Sink。到着 λ=0.8（指数, 平均間隔1.25）、
  サービス μ=1.0（指数, 平均1.0）。利用率 ρ=λ/μ=0.8。
- 理論値: `Lq=ρ²/(1-ρ)=3.2`, `Wq=Lq/λ=4.0`, `W=Wq+1/μ=5.0`, `L=λW=4.0`。
- 自己検査の実測: **util 0.798（理論0.800）, Lq 3.044（理論3.200）, Wq 3.797（理論4.000）**。
- 見どころ: ρ が1に近いほど行列が急増する非線形性。Queue の `Lq/Wq` を実行中に観察。

### 6.2 直列ライン＋バッファ（ボトルネック）（`serial_line_buffer`）
- 構成: Source → Q1→P1(1.0) → Q2→P2(2.0) → Q3→P3(1.5) → Sink。到着は十分速く
  ライン飽和。
- 理論値: 定常スループットは **最遅工程** で決まる。`throughput = 1/max(stage) = 1/2.0
  = 0.500 個/単位時間（=1800個/時）`。ボトルネックは P2。
- 自己検査の実測: **thr=0.500（理論0.500）**。
- 見どころ: P2 の前（Q2）だけ在庫が溜まり、P2 の稼働率が最も高くなる（＝ボトルネック
  の見つけ方）。P2 を速くしない限りライン全体は速くならない。

### 6.3 AGV混雑（容量1の共有通路）（`agv_congestion`）
- 構成: 3本のラインが完成品をAGV隊（搬送者3台）に渡し、AGVは **容量1の共有エッジ
  A→B（長さ30, 速度5）** を通って Sink へ。容量1なので単線・FIFOで直列化します。
- 理論値/主張: 自由走行の1脚は `30/5=6.0`。**容量1にすると搬送が直列化し、平均滞留
  （リードタイム）が自由流より必ず増える**。`network.edge_capacities` を外すと非混雑の
  基準ケースになります。
- 自己検査の実測: **混雑 79.4 > 自由流 28.2**（別の混雑検査では lead 55.07 > 15.57)。
- 見どころ: 容量制約が「距離」ではなく「渋滞」でリードタイムを押し上げる様子。

### 6.4 Process Flow 物流（`processflow_logistics`）
- 構成: トークンが AGV を1台 acquire → 集荷点(15,0)へ travel → load(2.0) → 荷降し点
  (15,20)へ travel → 実 Sink へ unload(1.0) → AGV を release、という **タスク列**。
- 理論値: `cycle = leg1(15/5=3) + load(2) + leg2(20/5=4) + unload(1) = 10.0`、配送1件。
- 自己検査の実測: **measured=10.0000（expected 10.0000）, delivered=1**（誤差1e-6以内）。
- 実行方法: 4段目「▶ PF実行」で先頭の登録 Process Flow を隔離実行し、KPI
  `avg_cycle_time` を読みます（「🔍 PF検査」で spec の静的検査も可能）。
- 見どころ: 個々の移動・積み降ろしが加算されて配送サイクルになること。経路に容量1の
  エッジを足すと、この時間の上に渋滞が乗ります（`pf-travel-congest` 参照）。

> 残る6本（M/M/c、共有作業者、段取り替え、故障MTBF/MTTR、蓄積コンベヤ、
> 実験/最適化デモ）の理論値・期待KPIは [`samples/README.md`](samples/README.md) の
> 一覧表にまとまっています。

## 7. ヘッドレスで再現・検証する（1分）

画面なしで、全サンプルが理論値に一致することを自動検証できます。エンジンの自己検査には
**全10サンプルを読み込み→実行→期待値照合する `[samples]` マーカー** が含まれます。

```sh
B=/path/to/Godot_v4.7.1-stable_linux.x86_64   # 実際のバイナリパス

# 1) インポート（クリーンなら何も出ない or IMPORT_CLEAN）
"$B" --headless --path . --import

# 2) 自己検査（終了コード0、[samples] n=10 passed=10 ... all_pass=true を確認）
SIM_HEADLESS_TEST=1 "$B" --headless --path .
```

`[samples]` マーカーの出力例（実測が期待値に一致）:

```
[samples] n=10 passed=10 mm1(util 0.80~0.80 ok) mmc(util 0.75~0.75 ok)
serial(thr=0.500~0.500 ok) shared(thr=0.098~0.100 ok) setup(thr=0.1429~0.1429 ok)
breakdown(thr=0.900~0.900 ok) conveyor(occ=5 thr=0.050 ok) agv(cong79.4>free28.2 ok)
pf(cycle=10.000~10.000 ok) opt(best=3601 mono=true ok) all_pass=true
```

各モデル固有の理論検証マーカー（`[mm1] [mmc] [conv-accum] [agv-traffic] [agv-cap]
[pf-travel] [optimize] [cal-op]` など）も同じ自己検査で出力されます。

## 8. つまずいたら

- **サンプルを選んでも何も起きない**: ドロップダウン先頭「📚 サンプルを選択…」は
  プレースホルダです。実際のサンプル名を選んでください。読込後はコンソールに期待値が出ます。
- **実測が理論値からズレる**: 実行が短すぎ／warmup不足の可能性。`長さ` を大きく（例
  100000）し `warm` を設定、あるいは「🧪 実験」で反復し95%信頼区間で見てください。
  理論値が信頼区間に入っていれば整合です。
- **他人のモデルを開くとダイアログが出る**: スクリプトを含むモデルの安全確認です。
  信用できない場合は「無効化して読込」でコードを実行せず構造だけ復元します。

以上で一通りです。まずは **📚 サンプル → `mm1_queue`** を読み込み、コンソールの期待値を
見てから「▶ 開始」を押してみてください。
